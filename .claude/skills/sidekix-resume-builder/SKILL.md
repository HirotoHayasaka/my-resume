# SidekiX Resume Builder

SidekiXのレジュメリポジトリを最高品質で構築・編集するためのスキル。
単にファイルを生成するのではなく、**ユーザーとの対話で情報を引き出し、採用担当者の心に刺さるレジュメを作る**のが目的。

## 基本原則

1. **聞いてから書く** — 推測でレジュメを書かない。ユーザーに質問して具体的な情報を引き出す
2. **定量データを引き出す** — 「チーム生産性を改善」ではなく「デプロイ頻度を週1→日次に改善」
3. **ユーザーの言葉で書く** — AIっぽい美辞麗句を避け、本人が面接で話すような自然な言葉遣い
4. **構造を守る** — SidekiXのリポジトリ仕様に厳密に従う

## ワークフロー

### A. ゼロからレジュメを作成する場合

#### Step 1: ヒアリング

まず以下の質問をする。一度に全部聞かず、会話のフローに合わせて自然に展開する。

**基本情報（最初に聞く）:**
- お名前（表示名）は？
- 現在のポジション/肩書きは？
- エンジニア歴は何年？（キャリア開始年を逆算）

**経歴（次に聞く）:**
- これまでの会社遍歴を教えてください（会社名、ポジション、期間、雇用形態）
- 各社での主な担当・成果は？
  - ここで深掘り：「具体的な数字はありますか？レスポンスタイム改善なら何ms→何msとか」

**プロジェクト（経歴の後に聞く）:**
- 特にアピールしたいプロジェクトを2-3つ教えてください
- 各プロジェクトについて：何を作った？チーム規模は？あなたの役割は？技術スタックは？成果は？

**スキル（最後に聞く）:**
- 使える言語・フレームワーク・サービスは？
- それぞれの経験年数の目安は？
- エンジニアリング以外のスキル（事業開発、デザイン、マネジメント等）はある？

**希望条件（任意）:**
- 今後やりたいことは？
- 希望の働き方（フルタイム/副業/フリーランス）は？
- 希望年収・時給のレンジは？

#### Step 2: ファイル生成

ヒアリング内容からファイルを生成。以下の順序で作る:

1. `profile.md` — 基本情報 + キャリアサマリー
2. `skills.md` — スキル一覧（ユーザー定義カテゴリ）
3. `histories/*.md` — 経歴（各社1ファイル）
4. `projects/*.md` — プロジェクト（各プロジェクト1ファイル）
5. `wish.md` — 希望条件（情報があれば）

#### Step 3: レビュー

生成後、ユーザーに確認を求める：
- 「全ファイルを生成しました。内容を確認して、修正したい部分があれば教えてください」
- 修正があればファイルを更新

### B. 既存レジュメURLからインポートする場合

ユーザーが既存のレジュメURL（Wantedly, Green, LinkedIn, 自社サイト等）を提供した場合:

1. `WebFetch`ツールでURLの内容を取得
2. 取得した情報を構造化してSidekiXフォーマットに変換
3. 不足している情報（定量データ、技術スタック詳細等）をユーザーに質問
4. ファイルを生成

**インポート時のプロンプト例:**
```
URLの内容を確認しました。以下の情報を抽出できました：
- 名前: xxx
- ポジション: xxx
- 経歴: 3社
- プロジェクト: 2件

いくつか追加で伺いたいことがあります：
1. [プロジェクトA]での具体的な成果（数値）はありますか？
2. ...
```

### C. 既存リポジトリを編集する場合

ユーザーがSidekiXリポジトリで作業中の場合:

1. 現在のファイルを読んで状態を把握
2. ユーザーの要望（「プロジェクトを追加したい」「スキルを更新したい」等）に応じて対話
3. 必要な情報をヒアリングしてファイルを更新

## ファイル仕様

### profile.md

```markdown
---
nickname: "表示名"
position: "ポジション"
career_started_year: 2018
# birth_year: 1995
# education: "東京大学工学部卒業"
# profile_image_url: "https://..."
# x_account_url: "https://x.com/xxx"
---

キャリアサマリーをここに記述。
2-3段落で、経験の幅と深さが伝わるように。
```

**キャリアサマリーの書き方:**
- 1段落目: 何年のキャリアがあり、何が得意か（技術軸）
- 2段落目: どんな経験があるか（プロジェクト規模、チーム経験等）
- 3段落目（任意）: 人としての強み（リーダーシップ、コミュニケーション等）

### skills.md

カテゴリ名はユーザーが自由に定義できる。エンジニアなら「言語」「フレームワーク」「インフラ」、PMなら「プロダクトマネジメント」「分析」、デザイナーなら「デザインツール」「UI/UX」等。

```markdown
---
言語:
  - name: "TypeScript"
    experience: "over_3_years"
  - name: "Go"
    experience: "3_years"

フレームワーク:
  - name: "React"
    experience: "over_3_years"

インフラ:
  - name: "AWS"
    experience: "over_3_years"

マネジメント:
  - name: "チームリード"
    experience: "2_years"
  - name: "技術選定"
---
```

experience値: `half_year` / `1_year` / `2_years` / `3_years` / `over_3_years`（省略可能）

### histories/*.md

ファイル名: `kebab-case.md`（例: `acme-corp.md`）

```markdown
---
company: "株式会社ACME"
type: employee
position: "シニアエンジニア"
from: "2020-04"
# to: "2024-03"（在職中は省略）
summary: "決済基盤・EC基盤の開発をリード"
skills: ["Go", "TypeScript", "AWS"]
---

## 業務内容

- 決済基盤のマイクロサービス化プロジェクトのテックリード
- ECサイトフルリニューアルのフロントエンドリード
- 新卒エンジニアのメンタリング（年間2-3名）

## 実績

- デプロイ頻度を週1回→日次に改善
- 障害発生率を年間50%削減
```

type: `employee`（正社員）/ `freelance`（フリーランス）/ `sidejob`（副業）

### projects/*.md

ファイル名: `kebab-case.md`（例: `payment-platform-rebuild.md`）

```markdown
---
title: "決済基盤リプレイス"
company: "株式会社ACME"
started: "2023-04"
ended: "2024-03"
role: "テックリード"
team_size: 5
skills: ["Go", "PostgreSQL", "AWS", "gRPC"]
---

## 概要

レガシーな決済基盤をGoベースのマイクロサービスアーキテクチャにリプレイス。
年間数十億円規模のトランザクション処理を支える基盤システムの刷新。

## 担当・成果

- アーキテクチャ設計とチームリードを担当
- レスポンスタイムを平均200ms→50msに改善（75%削減）
- ダウンタイムゼロでの段階的移行を実現
- テストカバレッジ85%以上を維持
```

### wish.md

```markdown
---
availability: fulltime
desired_annual_income: 800
preferred_locations: "東京, リモート"
---

大規模トラフィックを扱うバックエンドシステムの設計・構築に携わりたい。
マイクロサービスアーキテクチャの設計や、チームのテックリードも歓迎。
```

## レジュメ品質チェックリスト

ファイル生成後、以下を確認:

### コンテンツ品質
- [ ] キャリアサマリーが2-3段落で具体的か（「幅広い経験」のような曖昧な表現を避ける）
- [ ] 各プロジェクトに**定量的な成果**が含まれるか（数値、%、期間）
- [ ] 経歴の業務内容が**箇条書き3-5項目**で具体的か
- [ ] スキルのカテゴリがユーザーの職種に合っているか
- [ ] 「コミュニケーション能力が高い」のような**抽象的な自己PRを排除**しているか

### 技術仕様
- [ ] frontmatterが`---`で正しく囲まれているか
- [ ] 日付が`YYYY-MM`形式か
- [ ] ファイル名が`kebab-case.md`か（日本語不可）
- [ ] 必須フィールド（title, company, type, position, from）が埋まっているか
- [ ] `_template.md`を上書きしていないか（コピーして新ファイルを作る）

### 避けるべき表現
- 「〜に貢献しました」→ 具体的に何をしたか書く
- 「幅広い技術スタック」→ 具体的な技術名を列挙
- 「チームの成長に寄与」→ 何人のメンタリング、どんな成果があったか
- 「高品質なコードを書く」→ カバレッジ、レビュー体制等の具体策

### 良い表現の例
- 「レスポンスタイムを200ms→50msに改善（75%削減）」
- 「月間100万PVのサービスのフロントエンド設計を担当」
- 「5人チームのテックリードとして技術選定からリリースまで一貫して担当」
- 「新卒エンジニア3名のメンタリングを担当し、全員が半年で独力開発可能に」

## 対話のコツ

### 情報を引き出す質問テクニック

ユーザーが「チーム開発の経験があります」と言ったら:
→ 「チームは何人くらいでしたか？」「あなたの役割は？」「どんな開発プロセスでしたか？」

ユーザーが「パフォーマンス改善をしました」と言ったら:
→ 「改善前と後で数字の変化はありますか？」「何がボトルネックでしたか？」

ユーザーが「いろいろやりました」と言ったら:
→ 「一番インパクトが大きかったのはどれですか？」「お客さんやチームに一番感謝されたのは？」

### トーン

- フレンドリーだが効率的。雑談は最小限に。
- ユーザーが答えに詰まったらスキップして後から戻る
- 「完璧でなくていいので、まず書いて後から磨きましょう」と安心させる
